## 1. 项目概述

**QuantRust** 是一个专为A股市场设计的高性能、专业级量化交易平台。本项目旨在结合 **Rust** 后端的极致性能与安全性，以及 **React** 前端的丰富交互体验，为个人量化投资者、小型投资团队及金融学习者提供一套完整、高效且用户友好的端到端解决方案。

本文档将作为项目的核心设计蓝图，详细阐述从产品构想到技术实现的完整方案。

## 2. 产品设计与功能规划

我们的核心目标是打造一个覆盖策略研发、回测、模拟及实盘交易全生命周期的无缝体验。

### 2.1 产品愿景

成为国内领先的、以技术驱动的个人量化交易解决方案，让复杂的量化投资变得简单、高效和可靠。

### 2.2 目标用户

我们的目标用户群体具有多样性，其核心需求围绕着策略验证、执行效率和学习实践。

| 用户群体 | 特征与核心需求 |
| :--- | :--- |
| **个人量化爱好者** | 具备编程和金融知识，寻求功能强大、易于上手的工具来验证和执行交易想法。 |
| **小型私募/投资团队** | 需要稳定、高效、可扩展的系统来管理多策略多账户，对交易速度和风控有高要求。 |
| **金融/计算机专业学生** | 希望通过现代化平台学习和实践量化知识，需要丰富的学习资源和活跃社区。 |

### 2.3 核心功能规划 (MVP)

我们将采用敏捷迭代的方式，MVP阶段将聚焦于打通核心交易流程。

> 详细的产品功能点、非功能性需求及未来路线图，请参阅独立的 **[产品需求文档](./product_requirements.md)**。

## 3. UI/UX 设计方案

### 3.1 设计理念

界面设计将遵循**直观、数据密集、高响应性**的原则，借鉴 TradingView 和 QMT 等成熟平台的优点。默认采用**暗色主题**，以减少用户在长时间交易中的视觉疲劳，同时提供可切换的亮色主题。整体布局将采用模块化设计，允许用户进行一定程度的自定义。

### 3.2 界面布局设计

主仪表盘界面被划分为多个功能区域，以优化工作流程并确保关键信息一目了然。

![UI 布局线框图](./diagrams/ui_layout.png)

*   **顶部导航区**: 提供到系统所有核心模块的快速入口。
*   **图表核心区**: 用于行情分析的K线图表，是界面的中心。
*   **市场概览区**: 包含自选列表和深度盘口，提供实时的市场快照。
*   **账户交易区**: 用于跟踪持仓、管理订单和接收系统通知的面板。

## 4. 系统架构设计

### 4.1 总体架构

后端采用基于 **Rust** 的**事件驱动微服务架构**。该设计确保了低延迟、高吞吐和出色的容错能力，是构建金融交易系统的理想选择。

![系统整体架构图](./diagrams/architecture.png)

### 4.2 技术选型

我们选择了一套现代化、高性能的技术栈来支撑整个系统。

| 组件 | 技术选型 | 核心理由 |
| :--- | :--- | :--- |
| **语言** | Rust | 极致性能、内存安全、强大的并发能力。 |
| **Web框架** | Axum | 基于 Tokio 和 Hyper，性能卓越，与 Tower 生态无缝集成。 |
| **消息队列** | NATS | 轻量级、高性能，完美契合服务间的异步通信需求。 |
| **数据库** | PostgreSQL, ClickHouse, Redis | 分别用于业务数据、时序行情数据和实时缓存。 |
| **Python集成**| PyO3 | 在 Rust 中无缝调用 Python 代码，实现用户自定义策略。 |

### 4.3 后端模块化设计

项目后端代码通过 **Cargo Workspace** 组织，划分为多个独立的微服务和共享库，实现了高度的模块化和代码复用。

![Rust 后端模块架构图](./diagrams/rust_modules.png)

> 关于技术选型的详细论证、各微服务职责以及数据库表结构设计，请参阅 **[后端系统架构设计文档](./backend_architecture.md)**。

## 5. 数据流与实时监测

### 5.1 事件驱动数据流

系统通过一个响应式的数据流模型运作。市场数据、交易信号和订单状态作为事件在系统中传播，确保了数据的一致性和系统的弹性。

![事件驱动数据流图](./diagrams/dataflow.png)

### 5.2 实时监测架构

我们集成了一套全面的监控方案（Prometheus, Grafana, Loki, Tempo），以提供对系统健康状况和性能的深度实时洞察。

![监控系统架构图](./diagrams/monitoring_architecture.png)

> 关于事件的具体定义、数据管道的实现细节以及关键监控指标（KPIs），请参阅 **[数据流与实时监测方案文档](./dataflow_monitoring.md)**。
