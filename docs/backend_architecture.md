# A股量化交易工具- Rust后端系统架构设计

## 1. 架构概述

本系统后端将采用基于 **Rust** 语言的 **微服务(Microservices)** 架构，遵循 **事件驱动 (Event-Driven)** 的设计哲学。通过将不同业务领域的功能拆分为独立的、松耦合的服务，我们可以获得更高的系统可扩展性、可维护性和容错能力。服务间的通信将通过一个中心化的消息队列 (Message Queue) 进行，以实现异步解耦。

这种架构特别适合量化交易系统，因为它能够高效处理高并发的实时数据流，并能独立地扩展计算密集型任务（如回测）和I/O密集型任务（如行情接收）。

## 2. 技术选型

| 组件 | 技术选型 | 理由 |
| :--- | :--- | :--- |
| **编程语言** | **Rust** | 提供极致的性能、内存安全和强大的并发能力，是构建低延迟、高可靠性金融系统的理想选择。 |
| **Web框架** | **Axum** | 基于`tokio`和`hyper`，性能卓越，与`tower`生态系统无缝集成，便于添加中间件（如认证、限流）。 |
| **异步运行时** | **Tokio** | Rust社区最成熟、应用最广泛的异步运行时，提供了构建高性能网络服务的完整工具链。 |
| **消息队列** | **NATS** | 轻量级、高性能、易于部署的消息系统，其发布/订阅和请求/响应模式非常适合服务间的异步通信。 |
| **数据库 - 业务** | **PostgreSQL** | 功能强大、稳定可靠的关系型数据库，用于存储用户信息、策略代码、交易记录等结构化数据。 |
| **数据库 - 行情** | **ClickHouse** | 面向列的分析型数据库，对时间序列数据的存储和查询性能极佳，用于存储海量的历史K线和Tick数据。 |
| **数据库 - 缓存** | **Redis** | 高性能的内存数据库，用于缓存热点数据（如最新行情、用户持仓）和实现分布式锁等。 |
| **数据库交互** | **SQLx** | 异步的Rust SQL工具包，提供编译时查询检查，显著提升了数据库操作的安全性和健壮性。 |
| **Python集成** | **PyO3** | 允许在Rust中无缝地调用Python代码，是我们实现用户自定义Python策略的核心。 |
| **容器化** | **Docker** | 实现开发、测试和生产环境的一致性，简化部署和运维流程。 |

## 3. 模块化设计 (Cargo Workspace)

项目将采用Cargo Workspace进行管理，将后端系统划分为多个独立的Crate（包），每个Crate对应一个微服务或一个共享库。这种结构有助于代码复用和模块解耦。

![Rust后端模块架构图](diagrams/rust_modules.png)

- **`quantrust-gateway`**: API网关，作为系统的统一入口，处理所有HTTP和WebSocket请求，负责路由、认证、限流等。
- **`quantrust-core`**: 核心共享库，定义了系统中的基础数据结构（如`Order`, `Trade`, `Instrument`）、事件类型和全局配置。
- **`quantrust-data`**: 数据服务，负责从外部数据源（如AkShare, TuShare）摄取、清洗和存储行情数据。
- **`quantrust-strategy`**: 策略引擎，负责加载和执行用户的Python策略代码，通过PyO3与Python运行时交互。
- **`quantrust-backtest`**: 回测引擎，提供事件驱动和向量化两种回测模式，对历史数据进行策略回测。
- **`quantrust-trade`**: 交易引擎，管理订单生命周期，连接模拟交易和实盘券商接口。
- **`quantrust-risk`**: 风控引擎，在交易执行前进行风险检查，并实时监控账户风险敞口。
- **`quantrust-monitor`**: 监控服务，负责收集系统的日志、性能指标（Metrics）并提供健康检查接口。

## 4. 核心组件详解

### 4.1 事件驱动核心

事件驱动是本架构的核心。所有服务都是独立的，它们通过订阅和发布事件到NATS消息队列来进行通信，而不是直接相互调用。

![事件驱动数据流图](diagrams/dataflow.png)

**主要事件流：**
1.  **行情流**: `数据服务`获取行情后，发布`MarketDataEvent`。
2.  **信号流**: `策略引擎`监听到行情事件，执行策略逻辑后，可能产生`SignalEvent`。
3.  **订单流**: `风控引擎`监听到信号，进行检查后，发布`OrderEvent`。
4.  **成交流**: `交易引擎`监听到订单，执行后，发布`FillEvent`，并通知相关服务更新状态。

### 4.2 策略引擎与Python集成

`quantrust-strategy`服务是策略执行的核心。它会为每个运行的策略创建一个独立的沙箱环境。

- **Python沙箱**: 利用`PyO3`，策略引擎会启动一个嵌入式的Python解释器。
- **策略API**: Rust会向Python环境注入一个API模块，提供`get_data()`, `place_order()`, `get_position()`等函数，让Python策略代码可以与系统其他部分交互。
- **状态管理**: 策略的持仓、变量等状态会由Rust侧进行管理和持久化，确保策略在重启后能够恢复状态。

## 5. 数据库设计（高阶）

- **`users`**: 存储用户信息，包括加密后的密码和API密钥。
- **`strategies`**: 存储用户保存的策略代码和版本信息。
- **`backtest_reports`**: 存储每一次回测的结果摘要和详细交易记录的关联。
- **`trades`**: 记录所有成交回报，包括模拟交易和实盘交易。
- **`stock_candles_1m` (ClickHouse)**: 存储所有A股的1分钟K线数据，以`（股票代码，时间）`作为主键/排序键。
- **`stock_ticks` (ClickHouse)**: （未来扩展）存储Level-2的逐笔成交数据。

这个架构设计旨在平衡性能、可扩展性和开发效率，为构建一个专业、可靠的量化交易平台打下坚实的基础。
